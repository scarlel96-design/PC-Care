# PC 케어 파일·폴더 선택기 아키텍처

## 확인된 기존 문제

- 보안 금고와 업데이트 기능이 `Windows.Storage.Pickers`와 `InitializeWithWindow`를 각각 중복 구현했다.
- 보안 삭제는 `FileOpenPicker.PickSingleFileAsync()`만 호출하여 UI 문구와 달리 폴더 및 다중 파일을 지원하지 않았다.
- 선택 취소, 소유 창 부재, 초기화 실패가 모두 nullable 경로로 합쳐져 사용자가 버튼 무반응으로 인식할 수 있었다.
- 보안 삭제 페이지가 정적 `App.Shell`의 HWND를 직접 사용했고, 선택기 예외가 `async void` 이벤트 밖으로 전파될 수 있었다.
- 업데이트 선택기만 별도 WinForms 폴백을 사용하여 기능별 정책이 달랐다.

앱은 `net10.0-windows10.0.26100.0`, Windows App SDK `2.2.0`, WinUI 3, 비패키지형(`WindowsPackageType=None`), `asInvoker`로 확인했다. 관리자 권한은 앱 기본 실행이 아니라 설치·복구·업데이트 적용 등 명시 기능에서만 `runas`로 요청한다.

## 변경 후 구조

`IPathPickerService`가 단일 파일, 다중 파일, 폴더, 저장 위치 선택을 제공한다. 기본 구현 `PathPickerService`는 `Microsoft.Windows.Storage.Pickers`와 현재 창의 `AppWindow.Id`를 사용한다. 단일 창 공급자는 `IWindowProvider`로 캡슐화했고 페이지는 현재 메인 창을 명시적으로 전달한다.

모든 호출은 다음 결과를 구분한다.

- `Success`: 검증 가능한 경로 반환
- `Cancelled`: 사용자 취소 또는 취소 토큰
- `Failed`: HRESULT, 비식별 추적 ID, 사용자 메시지 반환

실패 진단에는 기능명, 백엔드, 동작, HRESULT, 예외 타입, 관리자/패키지 상태, OS·Windows App SDK 버전, WindowId/UI 스레드 상태, UTC·로컬 시간과 제한된 호출 스택을 기록한다. 사용자 전체 경로와 파일명은 기록하지 않는다.

## 기능별 흐름

- 보안 금고: 공통 선택기 → 파일/폴더 존재 및 유형 검증 → 금고 저장소·재분석 지점 차단 → 기존 금고 Import 파이프라인
- 보안 삭제: 다중 파일 또는 폴더 선택 → `SecureDeleteTargetSet` 정규화 → 부모·자식/대소문자/후행 구분자 중복 제거 → Dry Run → 명시 확인 → 취소 가능한 삭제 실행
- 업데이트: 공통 단일 파일 선택기 → `.spdup`/`.zip` 필터 → 기존 패키지 검사 및 적용 파이프라인

선택 중에는 관련 버튼을 비활성화하고 `PickerOperationGate`로 중복 선택기를 막는다. 성공·취소·실패 후 버튼과 포커스를 복원하며 상태 영역은 스크린 리더 Live Region으로 노출한다.

## 보안 삭제 정책

다음 대상은 실패 폐쇄 방식으로 차단한다.

- 드라이브 루트, Windows, System32, SysWOW64, Program Files, ProgramData, Recovery, 시스템 예약 계열
- 사용자 프로필·바탕 화면·문서 루트
- PC 케어 설치 디렉터리, 실행 파일, 사용자 데이터·로그, 금고 저장소, 현재 작업 디렉터리
- UNC·네트워크·장치 경로, 정션·심볼릭 링크·재분석 지점

선택한 폴더 내부 열거는 선택기 이벤트에서 하지 않고 Dry Run/삭제 단계의 백그라운드 작업에서 수행한다. 폴더 삭제는 재분석 지점을 따라가지 않으며 링크 자체만 정리한다. SSD/NVMe 파일 단위 삭제는 웨어 레벨링, 오버 프로비저닝, TRIM, 컨트롤러 매핑, 저널, 스냅샷, 클라우드·백업 사본 때문에 물리 셀 삭제를 보장하지 않는다고 UI와 결과에 고지한다.

보호 경로는 안전성 판정 직후 차단 결과로 확정한다. 차단된 드라이브 루트·Windows·사용자 프로필에 대해 저장장치 프로파일링이나 디렉터리 크기 계산을 수행하지 않으므로 Dry Run이 전체 디스크를 불필요하게 순회하지 않는다.

## 전역 예외 정책

선택기 오류는 서비스와 각 Task 경계에서 처리한다. 사용자 취소만 복구 가능한 전역 예외로 처리하고, 그 밖에 전역 처리기까지 도달한 예외는 오류 기록 후 치명적 상태로 분류하여 더 이상 `Handled=true`로 조용히 삼키지 않는다.

## 검증 범위

자동 테스트는 결과 상태, 사전 취소, 소유 창 부재, 중복 진입 방지, 파일/폴더 유형, 대소문자·후행 구분자 중복, 부모·자식 축약, 존재하지 않는 대상, 보호 경로, UNC·장치 경로, 재분석 지점을 다룬다. 실제 Windows 선택기 UI의 일반 권한·관리자 권한 수동 매트릭스는 릴리스 후보 바이너리에서 별도로 기록해야 한다.
